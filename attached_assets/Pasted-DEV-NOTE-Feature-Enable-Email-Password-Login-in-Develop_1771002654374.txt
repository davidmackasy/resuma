DEV NOTE
Feature: Enable Email + Password Login in Developer Mode Only
Scope: Development environment only
Do not affect production
Do not modify existing Google authentication in production
Do not break Stripe, auth sessions, or user roles

GOAL

Allow developers to sign in using email and password when the app is running in developer mode.

Production must continue using Google authentication only.

This must be environment-isolated and must not change production behavior.

ENVIRONMENT DETECTION

Use strict environment flag detection.

If using Node:
process.env.NODE_ENV

If using custom env flag:
process.env.APP_ENV

Define clearly:

Development:
NODE_ENV = "development"

Production:
NODE_ENV = "production"

Never rely on frontend-only checks.
Environment logic must be validated on backend as well.

AUTH STRATEGY ARCHITECTURE

Current Production Behavior

Google OAuth only

No email/password

Existing flow must remain untouched

Development Behavior

Enable email/password login

Keep Google optional if needed

Do not require Google in development

Implementation Rule

If NODE_ENV === "development"
→ enable email/password auth provider

If NODE_ENV === "production"
→ disable email/password auth provider
→ only allow Google OAuth

DATABASE ADJUSTMENTS

If using Supabase or custom auth table

Ensure users table supports:

id

email

password_hash (nullable in production)

auth_provider enum:

"google"

"email"

Rules

Development:
Allow auth_provider = "email"

Production:
Reject new users where auth_provider = "email"

Do not delete existing schema
Only extend safely if needed

BACKEND CHANGES

Add conditional provider loading

Example logic structure

if (process.env.NODE_ENV === "development") {
enableEmailPasswordAuth()
}

if (process.env.NODE_ENV === "production") {
disableEmailPasswordAuth()
}

Critical

Do NOT remove Google provider
Do NOT modify Google callback routes
Do NOT modify JWT structure
Do NOT modify session cookie format

Email/password auth must use same session system as Google
No duplicate auth systems

FRONTEND CHANGES

On Login Page

If development:
Show

Email input

Password input

Login button

Optional Google button

If production:
Show only Google button

Implementation

Wrap email/password form in conditional render

Example

{isDev && <EmailPasswordForm />}

Do not modify Google button UI
Do not change production UI styling

PASSWORD SECURITY

Use secure hashing

bcrypt or argon2

Never store plain text passwords

In development mode, hashing must still be active
No shortcuts

ROUTE PROTECTION

Ensure middleware handles both auth types

Auth middleware must:

Validate JWT/session

Validate user exists

Not care about provider type

Do not create separate session handling for email users

PRODUCTION SAFETY GUARD

Add backend hard block

If NODE_ENV === "production" and auth_provider === "email"

Reject login attempt

This ensures email login can never accidentally work in production

TESTING CHECKLIST

Create user with email/password

Login works

Session persists

Protected routes accessible

Stripe logic unaffected

Role system unaffected

In production

Email/password form does not render

API rejects email login

Google login works

Existing users unaffected

DO NOT TOUCH

Stripe webhook logic

Payment upgrade logic

Existing Google OAuth config

Admin roles

Credit system

Any production environment variables

Auth providers:

Email/Password

Optional Google

Auth providers:
